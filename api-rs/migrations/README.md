# api-rs migrations

`0001_initial_schema.sql` is the full Honcho schema, ported 1:1 from the Python
Alembic migrations (`../../migrations/versions/`, 25 revisions). It is embedded
into the `honcho-api-rs` binary at compile time via `sqlx::migrate!("./migrations")`
and applied on startup (see `src/main.rs`), replacing the Python
`scripts/provision_db.py` provisioner.

The migration is **schema-relative**: it carries no `public.` qualifiers, so the
objects are created in whatever schema the connection's `search_path` points at.
The app sets `search_path` from `DB_SCHEMA` in the pool's `after_connect` hook,
so this honours `DB_SCHEMA` exactly like the Alembic provisioner did.

## Verifying parity

The file was generated by snapshotting the real Alembic head and was verified to
produce a **byte-identical** schema: applying these 25 Alembic migrations and
applying `0001_initial_schema.sql` to a fresh database yield the same
`pg_dump --schema-only` output.

## Regenerating after a new Alembic migration

When a new Alembic revision lands on the Python side, regenerate this file (do
not hand-edit it):

```bash
# 1. Start a throwaway pgvector Postgres
docker run -d --name honcho-schema-snap -e POSTGRES_HOST_AUTH_METHOD=trust \
  -p 5440:5432 pgvector/pgvector:pg16
until docker exec honcho-schema-snap pg_isready -U postgres -d postgres; do sleep 1; done

# 2. Apply the Alembic head (from the repo root)
DB_CONNECTION_URI="postgresql+psycopg://postgres@localhost:5440/postgres" \
  DB_SCHEMA=public .venv/bin/python scripts/provision_db.py

# 3. Snapshot the schema
docker exec honcho-schema-snap pg_dump -U postgres -d postgres --schema-only \
  --no-owner --no-privileges --no-comments --exclude-table=alembic_version \
  > /tmp/honcho_schema.sql

# 4. Post-process into a schema-relative sqlx migration:
#    - drop \restrict/\unrestrict psql meta-commands
#    - drop the pg_dump SET / set_config('search_path','') preamble
#      (is_local=false GUCs would poison the pooled connection)
#    - drop -- comment lines
#    - CREATE EXTENSION ... WITH SCHEMA public  ->  CREATE EXTENSION ...
#    - strip every `public.` schema qualifier (safe: no 'public' string literals)
#    Then write api-rs/migrations/NNNN_<description>.sql
#
# 5. Verify byte-identical parity (apply both to fresh DBs, diff pg_dump output)
# 6. docker rm -f honcho-schema-snap
```

Bump the numeric prefix for each new migration; never edit an already-shipped
file (sqlx checksums applied migrations).
