# Plan: External Self-Hosted Honcho For Athena, With Agent-Led Bootstrap

## Goal
Make Athena easier to operate as a solo developer by keeping Honcho as a separately deployed memory service, then making Athena easy to point at that service with either:
- a copy-paste prompt that tells the agent to configure itself, or
- a later lightweight non-interactive CLI/bootstrap helper

This avoids bundling more infrastructure into Athena than necessary and keeps a later path open for GitHub-driven build/publish/deploy automation.

## Updated Recommendation
Do **not** lead with bundling Honcho into Athena's Docker Compose.

Recommended path now:
- deploy Honcho independently
- point Athena at it using the existing self-hosted Honcho provider support
- standardize a small, repeatable bootstrap flow that Athena can perform for itself
- document an OpenRouter-first Honcho config, since OpenAI is not your primary provider
- defer GitHub build/deploy automation until after the external-service workflow is smooth

## Why This Is Better Than Bundling
- Athena already supports self-hosted Honcho through `baseUrl`; no compose coupling is required to get value
- keeping Honcho separate gives cleaner failure isolation, upgrades, rollbacks, and backups
- it avoids making every Athena deploy drag along Postgres, Redis, deriver, and Honcho API lifecycle concerns
- it matches your stated operator model better: deploy Honcho once, then have Athena connect to it
- it is the lowest-effort path for a single developer managing multiple systems at once

## Provider Reality Check: OpenCode Go vs OpenRouter
Recommended operator stance:
- keep **Athena** on whatever model providers you already prefer, including OpenCode Go and OpenRouter
- configure **Honcho** separately, using the provider path it actually supports best today

Important nuance:
- Athena supports OpenCode Go as an agent/provider option
- Honcho's server-side LLM config is structured around `openai`, `anthropic`, and `gemini` transports
- for your current setup, the safest default for Honcho is **OpenRouter through the OpenAI-compatible transport**, not OpenCode Go directly unless OpenCode Go exposes a compatible API endpoint you explicitly want to route through

So the default plan should assume:
- Athena may use `OPENCODE_GO_API_KEY` and/or `OPENROUTER_API_KEY`
- Honcho should use `OPENROUTER_API_KEY` with OpenAI-compatible base URL overrides

## Database Decision: Share Existing ParadeDB

Your self-hosted ParadeDB PostgreSQL instance can be shared with Honcho — no need to spin up a separate database.

### Technical compatibility
- ParadeDB includes `pgvector`, which is the **only** PostgreSQL extension Honcho requires (`CREATE EXTENSION IF NOT EXISTS vector` at `src/db.py:74`).
- Honcho uses standard PostgreSQL FTS (`websearch_to_tsquery`), **not** ParadeDB's `pg_search` BM25 index. The extensions do not interfere.
- Honcho's own `pg_trgm` usage (trigram similarity) also has no ParadeDB conflict.

### Isolation via schema
Set `DB_SCHEMA=honcho` (not `public`) to keep Honcho's tables in their own namespace:

```bash
DB_CONNECTION_URI=postgresql+psycopg://your-user:your-pass@your-paradedb-host:5432/your-db
DB_SCHEMA=honcho
```

Honcho runs `CREATE SCHEMA IF NOT EXISTS` on startup, so no manual schema creation is needed.

### Resource considerations
Honcho adds query load from the deriver, dreamer, and hybrid retrieval (semantic + FTS fusion).
For solo developer workloads this is unlikely to matter. If it later becomes a bottleneck, extract Honcho to its own PostgreSQL via Alembic — no code changes required.

### Convention
- **Always share ParadeDB** when deploying Honcho — do not create separate database instances.
- Always set `DB_SCHEMA=honcho` to namespace tables away from other applications.
- Document the schema name in deployment notes so agents can verify against the correct namespace.

## Branch & Environment Convention

Four branches with a defined promotion flow:

| Branch | Role |
|--------|------|
| `main` | Integration — pull upstream changes here, sync into feature branches |
| `feat/<name>` | Feature work — branch off `main`, PR into `stable` |
| `stable` | Staging / dogfooding — merged features are tested here against stable Athena |
| `prod` | Production — `stable` is merged here once tests pass with no regressions |

Flow:
1. `feat/<name>` → PR into `stable`
2. Test on stable with stable Athena deploy
3. Once verified and regression-free → merge `stable` into `prod`
4. `main` periodically pulls upstream, then gets merged into active feature branches

Never commit directly to `stable` or `prod`. All changes go through `feat/` branches and PRs.

## Important Design Decisions

### 1. Bundled Honcho vs external Honcho
Recommended: external Honcho
- cleaner ops boundaries
- easier upgrades and rollbacks
- less coupling in Athena deploys
- already supported by Athena today

### 2. Prompt-only vs helper command
Recommended: prompt first, helper second
- prompt gives immediate leverage with almost no code changes
- helper command is the right next improvement once the exact bootstrap steps are stable

### 3. OpenAI vs OpenRouter for Honcho
Recommended: OpenRouter-first
- aligns with your real provider usage
- Honcho already has an OpenAI-compatible configuration path for proxy endpoints
- avoids making OpenAI a required part of your day-to-day stack

### 4. OpenCode Go as Honcho backend vs OpenCode Go only for Athena
Recommended: keep OpenCode Go as an Athena-side concern unless you explicitly confirm it exposes an OpenAI-compatible endpoint you want Honcho to hit
- safer and simpler now
- avoids speculative coupling to a provider shape Honcho may not expect directly
