# Plan: Choose or Fork Honcho vs LanceDB-pro for Athena Agents

## Goal
Pick the best external memory layer for Athena's agents, given Athena's current architecture and product direction.

## Key facts from the codebase
- Athena supports pluggable external memory providers, but only **one external provider can be active at a time**. Built-in `MEMORY.md` / `USER.md` stays active either way.
- Athena already has a dedicated **Honcho** memory plugin.
- Athena also has an **`athena-memory`** plugin, which is the fork's current pgvector-backed successor to the older LanceDB shim / LanceDB-pro path.
- Athena's stated direction emphasizes:
  - multi-tenant isolation
  - reliability
  - cross-session recall
  - user/agent continuity
  - Honcho-based dialectic user modeling

## Recommendation
If you must choose **one** for Athena's agents, choose **Honcho**.

If you are open to a fork, the best path is **not** "replace Honcho with LanceDB-pro". The better path is:

- **fork Honcho**
- keep its **peer / session / representation / dialectic / dreamer** model
- replace or extend its storage and retrieval substrate so it behaves more like your preferred self-hosted semantic stack

That is the most realistic way to get the best of both worlds.

## Why Honcho is the better fit
- Athena is not just doing document retrieval; it is trying to preserve **identity, preferences, working style, and relationship continuity** across sessions.
- Honcho is built around **peers, sessions, representations, summaries, and dialectic reasoning**, which matches Athena's agent architecture much better than a raw retrieval store.
- Athena's own docs describe Honcho as best for **multi-agent systems with cross-session context and user-agent alignment**.
- Honcho gives Athena a cleaner model for:
  - one user + many agent profiles
  - per-agent identity
  - shared user understanding
  - session summaries and just-in-time recall
  - agent-facing tools for profile, context, search, reasoning, and conclusion-writing

## Where LanceDB-pro / athena-memory is stronger
Use the LanceDB-pro lineage (`athena-memory`) when your main need is:
- self-hosted semantic retrieval
- lower vendor dependence
- local/controlled storage
- hybrid BM25 + vector recall over large text corpora
- a simpler L2 semantic memory layer rather than a social/user-modeling layer

It is stronger as a **retrieval substrate** than as a full agent-memory architecture.

## Why a Honcho fork is realistic
Honcho already has several pieces you would otherwise need to build from scratch:

- a first-class **peer/session memory model**
- **deriver / dialectic / dreamer** agent roles
- session summaries and working representations
- semantic search over messages and observations
- a **vector store abstraction** with support for:
  - `pgvector`
  - `lancedb`
  - `turbopuffer`

So the fork opportunity is not theoretical. Honcho already separates:
- **memory semantics** from
- **vector storage implementation**

That means you can keep the high-value agent-memory behavior and change the lower layer.

## Direct answer: can you self-host Honcho?
Yes.

Based on the current Honcho repo:
- Honcho documents a local/self-hosted setup using **Postgres + pgvector**
- it ships a `docker-compose` path for local database bootstrapping
- it can run with a local API server plus background deriver worker
- its vector layer is configurable as:
  - `pgvector`
  - `lancedb`
  - `turbopuffer`

Important nuance:
- upstream describes self-hosting primarily as suitable for **testing and evaluation**
- but technically, yes, it is a self-hostable open-source service and the architecture is not cloud-only

So your instinct is correct: if you self-host Honcho, you do **not** need to diverge from Honcho completely.

## What athena-memory was actually trying to build
`athena-memory` is not primarily competing with Honcho's peer/representation model. It is adding a stronger **retrieval and memory-operations pipeline** around a local-first store.

Core `athena-memory` ideas:
- hybrid **BM25 + vector** retrieval
- **RRF fusion**
- **cross-encoder reranking**
- **MMR diversity**
- score thresholds and length normalization
- explicit **multi-scope / multi-tenant filtering**
- pluggable **LanceDB or pgvector/ParadeDB** backend
- write-time **dedup**, chunking, and metadata extraction
- retrieval **observability** and CLI management
- optional **reflection** subsystem
- lifecycle rules like tiering, promotion/demotion, decay, compaction

This is much closer to a **retrieval-quality and memory-hygiene engine** than to a replacement for Honcho's dialectic memory model.

## What this means for the fork decision
The likely best move is:

- self-host **Honcho** first
- keep upstream Honcho concepts and APIs intact
- port only the **athena-memory capabilities Honcho actually lacks**

That is a much smaller and better-shaped project than building a separate Athena memory stack.

## Revised preferred strategy
### Stage 0 — Minimize divergence
- run self-hosted Honcho first
- verify whether its existing `pgvector` or `lancedb` modes already satisfy enough of the local/self-hosted requirement
- do not fork before this gap check

### Stage 1 — Add only missing retrieval features
If Honcho's current retrieval quality is the real gap, port from `athena-memory` in this order:
- BM25 + vector **hybrid retrieval**
- **RRF** merge
- reranking / MMR / score cutoffs
- query tracing and retrieval observability
- stronger tenant/scope filtering where Athena needs it

### Stage 2 — Add write-pipeline hygiene only if needed
- chunking for long memory entries
- dedup / compaction
- smart metadata extraction
- reflection store only if Honcho's existing summary/dreamer mechanisms are insufficient

### Stage 3 — Avoid semantic duplication
Do **not** import athena-memory features that duplicate Honcho's core value unless testing proves a real gap:
- peer/session identity model
- representations
- dialectic reasoning
- session summaries
- memory consolidation logic already covered by dreamer/deriver flows

## Likely mapping: athena-memory -> Honcho
### Good candidates to port into Honcho
- hybrid lexical + semantic retrieval
- pgvector/ParadeDB-first self-host defaults
- retrieval reranking and diversity
- retrieval stats / traceability
- stricter scope filters for Athena multi-tenancy
- optional compaction / dedup for stored observations or message-derived memory

### Probably unnecessary or lower priority
- separate memory-provider-shaped tool surface inside Honcho
- parallel lifecycle system that conflicts with Honcho's representation model
- reflection subsystem if Honcho summaries + dreamer already solve the need

## New recommendation
The new default recommendation is:

- **Yes, self-host Honcho**
- keep Athena's long-term memory direction centered on Honcho
- treat `athena-memory` as a source of retrieval and storage ideas to upstream into Honcho or carry in a shallow fork
- only fork once you identify concrete missing capabilities after running self-hosted Honcho locally

## Chosen first port priority
Prioritize **hybrid retrieval** first.

Reason:
- it directly targets the strongest Athena-memory differentiators
- it improves recall quality without rewriting Honcho's memory semantics
- it is the lowest-risk way to get "best of both worlds"
- it keeps the fork shallow and more upstream-friendly

## Hybrid retrieval port scope
First features to evaluate or port into Honcho:
- lexical + semantic dual retrieval
- BM25 / FTS alongside vector search
- RRF fusion of result sets
- reranking after fusion
- MMR diversity for final selection
- configurable score cutoffs and exact-match friendliness

## Hybrid retrieval success criteria
- better exact-name / exact-term recall than stock semantic search alone
- better mixed factual + conceptual recall on long histories
- no regression in Honcho's peer/session/dialectic flows
- tenant and session filters still apply correctly after fusion
- latency remains acceptable for Athena's operator workflows

## Revised recommendation
For Athena, the strongest long-term move is:

- **short term:** use Honcho as the active provider if you need results now
- **mid term:** fork Honcho and make it more self-hosted / Athena-native
- **do not** lead with a LanceDB-pro-first design unless retrieval is the only real goal

## Best-of-both-worlds architecture
### Keep from Honcho
- Workspace / Peer / Session / Message primitives
- Representation building
- Peer cards
- Dialectic reasoning API
- Dreamer-style consolidation
- Per-agent identity and shared user modeling

### Replace or adapt
- External storage backend selection
- Retrieval ranking strategy
- local-first deployment defaults
- Athena-specific tenancy, isolation, and profile mapping
- optional BM25 / hybrid recall behavior if your current semantic layer is better than stock Honcho defaults

## Preferred fork strategy
### Option A — Recommended
Fork **Honcho** and adapt its storage/retrieval internals.

Why this is best:
- preserves the API and concepts Athena already wants
- preserves the multi-agent peer model
- preserves dialectic reasoning and session summaries
- uses existing Honcho seams instead of inventing a new memory system
- gives you a local/self-hosted path without throwing away the higher-order memory logic

### Option B
Fork `athena-memory` / LanceDB-pro and try to add Honcho-like representations, peers, and reasoning on top.

Why this is weaker:
- you would be rebuilding the harder part
- retrieval is easier than durable user modeling
- you would likely end up recreating a subset of Honcho anyway

## What to fork inside Honcho
1. **Vector store defaults**
   - prefer `lancedb` or `pgvector` instead of managed-host assumptions
2. **Search fusion behavior**
   - tune hybrid retrieval to match Athena's real workloads
3. **Embedding provider choices**
   - make self-hosted / lower-cost embedding paths first-class
4. **Tenant/profile identity mapping**
   - align Honcho workspaces/peers with Athena's multi-tenant profile model
5. **Operational defaults**
   - Docker-first local deployment, simpler env surface, Athena-specific bootstrap

## What not to fork first
Do not start by rewriting Deriver, Dialectic, or Dreamer. Those are the hardest parts and the main reason to keep Honcho in the first place.

## Athena integration implication
Athena currently allows only one external memory provider at a time.

So the cleanest product shape is:
- Athena `memory.provider = honcho`
- but the selected Honcho instance is **your forked Honcho**, configured for local/self-hosted retrieval and Athena-friendly identity semantics

That is simpler than trying to teach Athena to run Honcho and `athena-memory` side by side.

It also means the best product story is probably:

- Athena users see one memory provider: **Honcho**
- your deployment decides whether that Honcho is:
  - upstream self-hosted as-is
  - upstream + config tuning
  - or a shallow Athena fork with better retrieval internals

## Forced-choice conclusion
- **Best for Athena's agents:** Honcho
- **Best for Athena's retrieval backend / self-hosted semantic layer:** `athena-memory` (LanceDB-pro lineage)

## Important constraint
Because Athena currently allows only one external memory provider at a time, the real decision is:
- Do you want **agent memory and personalization**? Choose Honcho.
- Do you want **local semantic recall infrastructure**? Choose `athena-memory`.

For the product described in Athena's README and fork intent, the first answer is the better match.

## Recommended operating model
### Default now
- Make **Honcho** the active external provider for Athena profiles that interact with users or coordinate other agents.
- Keep Athena's built-in memory and session search features enabled as the local baseline.

### Default later
- Keep `memory.provider` conceptually as **Honcho** in Athena
- move the implementation behind it to an Athena-owned Honcho fork with local-first vector storage and retrieval tuning

### Keep available as optional
- Keep `athena-memory` as the fallback / alternate provider for:
  - highly cost-sensitive deployments
  - strict self-hosting requirements
  - large internal corpora where retrieval quality matters more than user modeling

## Suggested Honcho configuration shape for Athena
- One shared **user peer** per real user
- One **AI peer per Athena profile/agent**
- Shared workspace for related profiles
- `recallMode: hybrid`
- Conservative starting cadence:
  - `contextCadence: 1`
  - `dialecticCadence: 2` or `3`
  - `dialecticDepth: 1`
- `sessionStrategy: per-repo` for coding agents, or per-chat/per-session where channel isolation matters
- Observation defaults should preserve tenant boundaries

## When to prefer athena-memory instead
Switch the active provider to `athena-memory` if any of these become the top priority:
- you want memory fully under your infrastructure with no Honcho dependency
- most value comes from retrieving notes, logs, docs, and prior transcripts rather than modeling people
- you need predictable storage/compute economics over richer dialectic behavior
- your agent's main failure mode is missing factual recall, not poor personalization

## Execution plan for a later implementation turn
1. Confirm the primary success criterion for the fork:
   - best agent personalization
   - strongest self-hosting
   - lowest cost
   - best retrieval quality
2. Stand up Honcho locally with its current self-hosted path.
3. Benchmark current Honcho against the specific Athena-memory goals:
   - hybrid recall quality
   - exact term lookup
   - long-history retrieval
   - tenant/profile isolation
   - local ops simplicity
4. Only after that, identify the missing features from athena-memory.
5. If gaps remain, fork Honcho and change storage/retrieval first, not memory semantics.
6. In that fork, prioritize the hybrid retrieval stack before any memory-hygiene or reflection work.
7. Keep Athena pointed at the forked Honcho provider.
8. Configure shared workspace + per-profile AI peers.
9. Start with low-cost cadence/depth settings.
10. Validate on five scenarios:
   - preference recall across sessions
   - profile isolation across Athena agents
   - tenant isolation under gateway/multi-profile use
   - retrieval quality on long operator history
   - operational simplicity of self-hosted deployment
11. Keep `athena-memory` available as the fallback retrieval-first provider, not the main default.

## Non-recommended path
Do not make `athena-memory` the default external provider for Athena if the goal is coherent long-term agent behavior. It solves retrieval better than it solves identity, social memory, or agent-user alignment.

## Open design decision
The main fork choice is now:

- **API-first fork:** preserve Honcho's API and Athena's existing plugin contract almost unchanged, only swap internals
- **Platform-first fork:** build a more Athena-specific memory service that borrows Honcho's ideas but diverges faster

For maintainability, the safer default is **API-first fork**.
